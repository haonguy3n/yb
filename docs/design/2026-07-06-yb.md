# yb — a Go, kas-compatible Yocto build orchestrator

*Design note — 2026-07-06*

## Why

We build Yocto images (e.g. `meta-irisentinel`, an i.MX6UL HAB-signed firmware)
with [kas](https://kas.readthedocs.io) + `kas-container`. kas works, but it is a
Python dependency chain wrapped in a shell script (`build.sh`) that bolts on the
things we actually need: our own build container, a shared host cache, HAB
signing keys, and an ssh deploy key. `yb` replaces that stack with a single
static Go binary that:

- reads our **existing kas `.yml` files unchanged** (drop-in), so real kas still
  works on the same tree and we can A/B them;
- **builds its own build container aligned to the Yocto release** named in
  `yb.yaml` (`version: zeus` → Ubuntu 18.04 + python2; `kirkstone` → 22.04), from
  an embedded Dockerfile — no prebuilt image to manage, no kas inside it;
- runs the build in that image, auto-mounting cache, HAB keys, and the ssh key;
- keeps everything "for our needs" in a separate `yb.yaml`, so the kas files stay
  pristine.

It is deliberately small: kas's core is *checkout repos to pinned refs →
generate `local.conf`/`bblayers.conf` → run bitbake*. That is a few hundred lines
of Go, no magic.

## Non-goals (v1)

- No re-implementation of every kas feature (menu/kconfig, multiconfig, repo
  patches, `kas dump`, signed manifests). Add on demand.
- Not a package builder. That is what `osb` is; `yb` orchestrates bitbake.

## Configuration — one kas file

### 1. kas keys — unchanged, parsed by yb

The subset of the kas schema we consume (everything `irisentinel.yml` +
`base.yml` use):

```yaml
header:
  version: 10
  includes: [base.yml]        # relative to this file; recursive; deep-merged
machine: irisentinel9x9
distro:  iritech-imx6ul
target:  [iritech-hab-firmware]   # scalar or list
repos:
  poky:
    url: https://git.yoctoproject.org/poky
    path: repos/poky              # checkout dir, relative to project root
    commit: d88d62c...            # commit (pinned) or branch — see note
    layers: { meta:, meta-poky: } # subdir layers; null value = enabled
  meta-freescale:
    url: https://github.com/Freescale/meta-freescale
    path: repos/meta-freescale    # no `layers:` → the repo root is the layer
    commit: 14f1a63...
  iritech:                        # no `url:` → the project dir itself
    layers: { meta-iritech:, meta-openssl-caam: }
local_conf_header:
  iritech: |
    ACCEPT_FSL_EULA = "1"
    ...
```

**Repo refs:** yb supports `commit:` (pinned) and `branch:` only — not kas's
legacy `refspec:` or `tag:`. Files using `refspec:` must migrate the field to
`commit:` (for a pinned SHA) or `branch:`.

**Merge semantics (must match kas):** for each file, load its `includes`
depth-first (base first), deep-merge, then the including file overrides. Maps
merge recursively; scalars and lists are replaced by the higher-priority file.
Consequence for `iritech.layers`: `base.yml`'s `{meta-iritech-test:}` and
`irisentinel.yml`'s `{meta-iritech:, meta-openssl-caam:}` **union** to all three
— same as kas.

**Layer path resolution:** a repo checks out at `dir` = `path` (or the repo name)
relative to the project root; a url-less repo is the project root itself. Its
layers are `dir/<layer-name>` for each enabled entry, or just `dir` when `layers`
is absent. These become container paths under `/work` (see below).

### 2. The `yb:` block — our orchestration, inside the kas file

Rather than a sidecar file, yb reads its orchestration from a `yb:` block in the
top-level kas file. kas ignores the key; yb parses it. The presence of the block
also marks that file as the buildable *entry* (auto-detected; `-f` overrides).

```yaml
# irisentinel.yml (alongside machine/distro/target/repos)
yb:
  version: zeus                     # Yocto release -> yb builds yb-yocto:zeus
  cache:   /srv/yocto-cache         # DL_DIR=<cache>/downloads, SSTATE_DIR=<cache>/sstate
  ssh_key: ~/.ssh/iri               # mounted read-only for private git
  mounts:                           # extra bind mounts (host path[:ro])
    - /srv/old-hab-keys/irisentinel:ro
  # image: my/prebuilt:tag          # optional override — skip building, use this
```

`cache` defaults to `/srv/yocto-cache`. Either `version` or `image` is required;
flags override the block.

### Version → image

`internal/image` maps a release name to a `Profile{Ubuntu, Extra}` and builds
`yb-yocto:<version>` from an embedded Dockerfile via `docker build -` (empty
context, Dockerfile on stdin), passing `UBUNTU_VERSION`, `EXTRA_PACKAGES`, and the
caller's `UID`/`GID` as build args. The image carries only the Yocto host deps +
locale + a non-root `builder` user + an ssh config — **no kas**. Built lazily and
cached; `--rebuild` forces it. `zeus` adds `python python-minimal` (python2) for
legacy i.MX BSP layers. `image:` bypasses this entirely.

## Pipeline

```
yb build [targets...]
  1. resolve entry kas file (-f, or the file with a `yb:` block)
  2. parse it, resolve includes, deep-merge -> Config (+ yb block, flag overrides)
  3. checkout each repo to its pinned ref (host git)        -> repos/
  4. write build/conf/local.conf + build/conf/bblayers.conf (container paths)
  5. ensure image yb-yocto:<version> (build if missing)
  6. docker run <image>: source poky/oe-init-build-env build && bitbake <targets>
```

- **Checkout is host-side** (`git` on the host). The public layers are https; our
  private layers live in the project repo already. When `ssh_key` is set, git runs
  with `GIT_SSH_COMMAND="ssh -i <key> -o IdentitiesOnly=yes"`.
- **Conf generation** writes container paths: the project bind-mounts to `/work`,
  so a layer at `<project>/repos/poky/meta` is written as `/work/repos/poky/meta`.
  `DL_DIR`/`SSTATE_DIR` point into the cache, which is mounted at the *same* path
  in the container as on the host, so paths agree.
- **`oe-init-build-env`** is the blessed entry to set `BUILDDIR`/PATH; it only
  copies sample confs when they are absent, so our pre-written confs survive.

### Generated `local.conf`

```
# Generated by yb — do not edit.
<local_conf_header entries, deterministic order>
MACHINE ??= "<machine>"
DISTRO  ??= "<distro>"
DL_DIR     ?= "<cache>/downloads"
SSTATE_DIR ?= "<cache>/sstate"
BB_NUMBER_THREADS ??= "<nproc>"
PARALLEL_MAKE     ??= "-j <nproc>"
```

### Generated `bblayers.conf`

```
# Generated by yb — do not edit.
POKY_BBLAYERS_CONF_VERSION = "2"
BBPATH = "${TOPDIR}"
BBFILES ?= ""
BBLAYERS ?= " \
  /work/repos/poky/meta \
  /work/repos/poky/meta-poky \
  ... \
  /work/meta-iritech \
"
```

## Commands

Deliberately small. yb runs from the project directory; there are no override
flags — version/cache/ssh_key/mounts all come from the `yb:` block.

- `yb build [file.yml ...] [targets...]` — the pipeline above. Positional
  `*.yml`/`.yaml` args are kas entry files (kas-style, `a.yml b.yml` or
  `a.yml:b.yml`, deep-merged left-to-right for CI overlays); other positionals
  are bitbake targets; with none, yb auto-detects the file with a `yb:` block.
  `--force` fetches and hard-checks-out the pinned ref (branch → latest origin
  tip), discarding local changes.
- `yb shell [file.yml ...]` — same environment, but `exec bash` instead of
  bitbake, for devshell/debugging inside the sourced build env.
- `yb version`.

## Layout (mirrors osb)

```
yb/
  cmd/yb/main.go        CLI dispatch
  internal/config/      kas yaml parse + include deep-merge + layer resolution
  internal/project/     yb.yaml load + defaults
  internal/image/       version->profile table, embedded Dockerfile, image build
  internal/repo/        host-side git checkout
  internal/conf/        local.conf + bblayers.conf generation
  internal/runner/      docker run for build / shell
  docs/design/          this note
```

## Open questions (post-v1)

- Multiple kas files on one CLI (`kas build a.yml:b.yml`)? kas unions them; we can
  extend `kas_file` to a list.
- `env:` passthrough and `bblayers_conf_header` (parsed but rarely used here).
- Auto-selecting the version from the poky ref/branch instead of `yb.yaml`.
